# 走马村「云脉寿岭·荔水走马」— 前端网页端与AIGC场景体验系统 PRD

> **系统定位**：把"看内容"变成"能理解、能预订、能认养、能复访"的 Web/PWA 平台。
> **核心原则**：先轻量化 → 再场景化 → 最后平台化。不做重景区复制，不做"炫技大屏"，以真实乡村资源为核心。

---

## 1. 产品目标

### 1.1 核心目标

把走马村四境（古道叙事境、荔田共生境、韧谷研学境、岭上共居境）、院落、门票、认养、天气、路线、反馈和支付串成完整转化闭环。

### 1.2 关键成功指标

| 指标类别 | 具体指标 |
|----------|----------|
| **认知类** | 四境内容页停留时长、路线生成使用率、云端体验完播率 |
| **交易类** | 院落预约完成率、门票预购转化率、支付成功率 |
| **关系类** | 荔枝树认养转化率、消息触达率、季节性复访率 |
| **运营类** | 反馈闭环时效、投诉率、人工客服转接率、内容更新时效 |

### 1.3 参考范式

- **阿那亚**：目的地层 + 运营层 + 内容层 + 社群层的信息架构（借鉴平台结构，非视觉风格）
- **拈花湾**："内容介绍 → 库存选择 → 日期选择 → 立即订购"的交易路径

## 2. 用户画像与关键旅程

### 2.1 用户画像

| 用户画像 | 主要诉求 | 典型入口 | 成功动作 | 产品注意点 |
|----------|----------|----------|----------|------------|
| **近郊周末游客** | 半日/一日微度假、轻松出片、快速预约 | 首页、活动海报、地图入口 | 路线生成、购票、院落轻量预约 | 页面要快、信息要直观 |
| **亲子家庭** | 食育、认养、采摘、低风险路线 | 荔田共生境、课程页、天气页 | 报名课程、认养树、购买伴手礼 | 明示时长、体力难度、雨天备用方案 |
| **研学/团建组织者** | 成套线路、讲解、课程、团体预约 | 韧谷研学境、古道叙事境 | 发起团体咨询、锁定档期、支付定金 | 需要报价单、人数梯度、联系人 |
| **院落住客** | 住一晚、看风景、体验村宴与晨间农事 | 岭上共居境、套餐页 | 预约院落、加购活动、评价 | 强调入住须知、停车与步行提示 |
| **认养用户** | 长周期陪伴、养护参与、收获权益 | 荔田共生境、认养页 | 认养下单、查看树档案、参与养护 | 要有成长记录与仪式感 |
| **村民/房东/运营协作者** | 发布活动、管理房源、处理反馈 | 运营后台 | 更新库存、上传养护记录、关闭工单 | 后台必须手机可用、步骤简单 |

### 2.2 四条关键用户旅程

1. **"内容浏览 → 路线决策 → 院落/门票成交 → 现场导览 → 反馈复访"**
2. **"荔枝树认养 → 成长追踪 → 养护参与 → 收获兑现 → 再传播"**
3. **"研学或活动咨询 → 档期确认 → 套餐下单 → 到场执行 → 口碑沉淀"**
4. **"村民/房东服务接单 → 内容更新 → 库存处理 → 反馈闭环 → 收益结算"**

| 层次 | 内容 | 阶段 |

```
├── 古道叙事境内容页
│   ├── 图文导览
│   ├── 音频导览
│   ├── AI 讲解摘要

├── 荔田共生境内容页

├── 院落预约页
│   ├── 日历库存

│   ├── 我的订单（门票/院落/认养）

│   ├── 微信支付/支付宝选择
```

### 4.2 交互主流程

```
首页/四境门户 → 看四境内容 → AI导览摘要/音频/云体验
                      ↓
              生成游览路线 ← 输入时长/人群/天气偏好
                      ↓
         高德路线 + 天气校验 + 模板匹配
                      ↓
              输出路线卡 + 推荐预约
                      ↓
    ┌────────┬─────────┼─────────┬────────┐
    ↓        ↓         ↓         ↓        ↓
预约院落  购买门票  认养荔枝树  创建订单   支付
    ↓        ↓         ↓         ↓        ↓
日历库存  日期数量   树档案+   订单中心  电子票/
入住须知  选择      认养协议   支付状态  入住码/
                      ↓                 认养证书
                   权益说明
                      ↓
              现场导览/反馈/分享/二次购买
```

### 4.3 设计原则

1. **路线生成必须可解释**：不只是一张地图，必须同时显示总时长、步行强度、是否适合老人/儿童、雨天备选方案和需要提前预约的节点。
2. **认养页面必须有"仪式感 + 信任感"**：展示树档案、成长时间线、养护记录、权益说明和收获兑现方式。
3. **后台与村民协作端必须尽量"低培训成本"**：考虑实际居住人口有限、老龄化较明显、运营主体不足的现实。

## 5. 技术架构

### 5.1 技术栈

| 层级 | 技术选择 | 说明 |
|------|----------|------|
| **前端** | Next.js + TypeScript | Web/PWA、SSR、SEO、移动端适配 |
| **后端/BFF** | NestJS | 账户、预约、票务、认养、反馈和运营接口 |
| **数据库** | PostgreSQL + PostGIS | 订单、内容、地理点位与路线模板 |
| **缓存** | Redis | 缓存、会话、幂等锁和任务状态 |
| **对象存储** | 对象存储 | 图片、视频、音频、数字证书与导览资料 |
| **地图** | 高德开放平台 | JS API、路径规划、天气查询、微信小程序插件 |
| **天气** | 和风天气 | 分钟级、逐小时、逐天预报、灾害预警、GeoAPI、多语言 |
| **支付** | 微信支付 + 支付宝 | 支付回调与退款流程 |
| **模型编排** | ModelProviderAdapter | 统一编排，不绑定单一供应商 |

### 5.2 模型层策略

| 供应商/模型 | 适用场景 |
|-------------|----------|
| OpenAI Responses API / GPT | 高质量创意、总结、多模态生成 |
| 百炼 / Qwen / DeepSeek / GLM | 国内替代、成本分层、合规分层 |
| Anthropic Claude | 严格 schema、工具调用边界 |

**核心原则**：抽象为 `ModelProviderAdapter`，把"导览摘要、路线文案、翻译、视觉物料、反馈总结"分配给不同模型。高频路线生成和反馈总结应尽量使用 mini 或国产模型，避免每个用户请求都触发联网搜索。

## 6. 权限模型

推荐至少分为八类角色（RBAC）：

| 角色 | 权限范围 |
|------|----------|
| 游客 | 浏览公开内容、生成路线、查看天气 |
| 注册用户 | 预约、购票、认养、反馈、查看个人订单 |
| 认养用户 | 注册用户权限 + 查看树档案、养护记录、参与养护活动 |
| 房东/院落协作者 | 管理房源库存、更新入住状态 |
| 内容编辑 | 管理四境内容、场景页配置 |
| 客服运营 | 处理反馈工单、查看订单 |
| 财务/订单管理员 | 订单管理、退款操作、对账 |
| 系统管理员 | 全局配置、权限管理、审计日志查看 |

**审计要求**：支付回调、树档案修改、价格变更、退款操作和删除用户数据都必须进入审计日志。

## 7. 隐私与合规

1. AI 导览、反馈总结、翻译、路线解释等场景，优先发送**脱敏后的结构化片段**，不把手机号、精确地理位置、未脱敏订单文本直接送到境外模型。
2. 公开用户问答场景，优先采用**国内模型网关**或最小化数据出境方案。
3. OpenAI API 默认生成最长 30 天的 abuse monitoring logs，`store=false` 不等于零保留。
4. 需同时考虑《生成式人工智能服务管理暂行办法》《个人信息保护法》《网络安全法》的边界。
5. **以上只构成产品设计建议，不替代专项法律意见。**

## 8. 时间线与成本

### 8.1 时间线

#### 小型团队（3-5人，6个月）


#### 中型团队（8-12人，4个月）

```

08/01 ─── 四境内容与路线引擎 / 预约票务与认养支付
09/01 ─── 运营后台与联调
10/01 ─── 测试试运营与上线
```

### 8.2 成本估算

| 实施规模 | 平均配置 | 开发周期 | 人力成本 | 云与API | 合计 |
|----------|----------|----------|----------|---------|------|
| **小型团队** | 产品0.5、设计0.5、全栈2、测试0.5、运维0.2 | 6个月 | 28-55万 | 4-12万 | **32-67万** |
| **中型团队** | 产品1、设计1、前端2、后端1.5、测试1、运维0.5 | 4个月 | 55-95万 | 6-18万 | **61-113万** |

### 8.3 月度运营成本

上线后月度运营成本（基础云资源 + 短信 + 地图 + 天气 + 支付通道 + AI 调用）：小型 2.5-8 万 RMB/月。

**成本控制要点**：
- 高频路线生成和反馈总结尽量使用 mini 或国产模型
- 避免每个用户请求都触发联网搜索
- 天气、路线模板、常见导览问答都应做缓存与复用

## 9. 后续演进

| 阶段 | 重点 | 内容 |
|------|------|------|
| **第一阶段** | 跑通闭环 | 首访转化 + 认养复访，让系统变成经营工具 |
| **第二阶段** | 增强体验 | 360 内容、轻 AR、数字讲解员、研学互动脚本、多语言 |
| **第三阶段** | 智能升级 | 生态感知、数字孪生、动态定价、果园传感器、策略推荐 |

**核心理念**：先把真实资源组织好，再把数字能力叠上去。避免"系统很复杂，但村庄没有真正因为它而更好运营"的常见失败路径。
