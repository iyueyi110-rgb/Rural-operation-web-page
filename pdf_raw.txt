后台⾃动化抓取与分析系统研究计划
执⾏摘要
后台系统的⽬标不是“每天抓⼀点⽹⻚然后⽣成⼀份漂亮 PPT”，⽽是建⽴⼀个可持续的数据运营底座：每天 ⾃动采集⾃有站点、预约票务、认养记录、农产品销售、游客反馈、天⽓和关键运营事件，做 ETL、出⽇报、 给出明⽇优化建议，并在异常时主动告警。项⽬材料⾥已经明确指出，AIGC 云脑应连通资源建档、游客体验、 ⽣态感知和经营复盘；同时也强调⾛⻢村不适合⼀开始做“重系统”。因此，后台系统的 MVP 应该优先聚 焦“⾃有数据闭环 + 半⾃动建议 + ⼈⼯审核发布”，⽽不是上来就尝试全⾃动化运营决策。 ﬁleciteturn ﬁle  ﬁleciteturn ﬁle 
⽬标与范围
后台系统的⽬标可以概括为五件事。第⼀，把分散源头统⼀登记。第⼆，把原始数据可靠⼊库。第三，把原始 数据转成可运营的指标表。第四，把指标表转成“明⽇优化建议”。第五，把异常和失败即时告警。只有这五 件事都成型，前台功能才会越⽤越聪明。ﬁleciteturn ﬁle 
在数据范围上，建议分成“必须抓、⾃有优先、审批后抓”三层。必须抓的是⾃有⽹⻚、院落和票务订单、认 养与养护⽇志、反馈⼯单、天⽓、活动排期与基础埋点；⾃有优先的是农产品销售、物流回传、课程报名、村 ⺠协作记录；审批后抓的是外部公开⽹⻚，如经确认可抓的活动⻚、媒体⻚、评价⻚和标杆案例⻚。这⾥的关 键判断是：农产品销售和游客体验，优先接⾃有订单系统、⽀付回调、⼆维码问卷、运营台账，不要把“屏幕 抓取第三⽅⻚⾯”当成主数据源。这样既准确，也更容易合规。ﬁleciteturn ﬁle 
建议把后台输出固定为三类。⼀类是⽇业务表，例如昨⽇访客、院落⼊住、票务预售、认养新增、投诉与表 扬、农产品销量。第⼆类是专题运营表，例如四境热度、活动转化、内容表现、⽓象影响。第三类是“明⽇优 化建议”，必须采⽤“指标证据 + 建议动作 + 责任⼈ + 预期影响 + 审核状态”的格式，⽽不是⼀段⼤⽽空的 AI 总结。这样最好接⼯单或待办系统，不让建议停留在⽇报⾥。ﬁleciteturn ﬁle 
详细功能清单与优先级
后台⾃动化系统建议采⽤“Bronze / Silver / Gold”三层思路：原始层保留抓取和导⼊证据，清洗层统⼀格式和 主键，指标层⽣成经营主题数据集。这个思路与 Airﬂow 的调度模型、dbt 的数据转换模型天然兼容。Airﬂow 官⽅⽂档提供 scheduler、webserver、CLI、官⽅ Docker image、Helm Chart 和⼤量 provider；dbt 则明确 ⽤于把 raw warehouse data 转为 trusted data products，并提供 lineage、tests、contracts 和 governance。 16

推荐的技术路线是“Airﬂow 编排 + 抓取器/导⼊器 + 原始数据区 + dbt 业务建模 + 指标服务 + 建议引擎 + 监控告 警”。Airﬂow 官⽅具备 Docker 与 Helm 部署路径，适合⼩中型团队快速落地；dbt 把 SQL 转换、测试、⾎ 缘、契约与治理放到⼀起，很适合 D+ 报表和可审计数据模型；Prometheus 把指标存成带标签的时间序列， Grafana 负责 dashboard 与 alerting。换句话说，这是⼀套⾏业⾥被⼴泛证明过的“可持续维护”的后台组 合，⽽不是⼀套过度定制、难以交接的脚本堆。 18
数据库建议分两层。交易与配置仍放 PostgreSQL；原始抓取⽂件、HTML、JSON、截图和导出⽂件进对象存 储。若前期⽇数据量不⼤，业务宽表也可先放 PostgreSQL；等到⾏为埋点、订单、⽇志达到更⾼规模时，再增 量引⼊ ClickHouse ⼀类分析库。这样能降低⾸期复杂度，同时保留扩展性。PostgreSQL ⽂档中已经覆盖⻆ ⾊、备份、HA、监控和本地化章节，⾜够⽀持初中期的主数据库能⼒。 19
模型调⽤侧建议延续前台的 provider adapter。原因有三。其⼀，百炼提供 OpenAI 兼容接⼝，并⽀持模型监 控；其⼆，DeepSeek API 直接兼容 OpenAI SDK 形态；其三，智谱提供⼯具调⽤、联⽹搜索、结构化输出与知 识库服务。这样后台的建议引擎、⽇报摘要和反馈聚类就能在“⾼质量模型、低成本模型、国产模型、离线模 型”之间弹性切换。 20
